# Auth Review — Issue #5 (`feat/authentication`)

## Summary

The auth design matches the decision log: Supabase for identity (Google, GitHub, magic link), app-issued JWT in memory, opaque refresh token in an HttpOnly cookie with rotation, backend-only credential handling, and `authLimiter` at 5/15min on `POST /api/auth/login`. Middleware behavior is correct and well tested. Frontend wiring (login, callback, AuthContext bootstrap, Nav logout, ProtectedRoute) covers the intended flows.

**Acceptance criteria**

| Criterion | Verdict |
|---|---|
| Google / GitHub OAuth | Met (`getOAuthUrl` + PKCE cookie + code exchange + login UI) |
| Magic link | Met (`sendMagicLink` + `token_hash` / hash `access_token` completion) |
| Session persists (auto-refresh from cookie) | Partial — refresh on mount only; concurrent rotation race can drop session; no mid-session JWT renewal |
| Protected routes redirect with `?redirect=` | Met in `ProtectedRoute` (component not yet used on a page) |
| Logout clears cookie + invalidates refresh | Met |
| Auth rate limit 5/15min | Met (`authLimiter` max 5, window 15m; not on `/refresh`) |
| `authMiddleware` rejects expired / missing / invalid | Met (+ unit tests) |
| `optionalAuthMiddleware` attaches user when present | Met in implementation (+ tests); not mounted on any route yet |

**Overall:** solid foundation with a few real bugs — open redirect, refresh-rotation races, and missing mid-session access-token renewal — that should be fixed before treating auth as production-ready.

## Issues

### Issue 1 -- Severity: bug
- File: `packages/frontend/app/auth/login/page.tsx:12`, `packages/frontend/app/auth/login/page.tsx:28`
- File: `packages/frontend/app/auth/callback/page.tsx:36`, `packages/frontend/app/auth/callback/page.tsx:46`
- Description: The post-login redirect target is taken from the query string (`?redirect=`) and/or `sessionStorage` (`bh_auth_redirect`) with no same-origin / path-only validation. Login uses `window.location.replace(redirect)`, which will navigate to absolute or protocol-relative URLs (e.g. `https://evil.example`, `//evil.example`). An attacker can craft `/auth/login?redirect=https://phish.example` and, after a successful OAuth/magic-link login, send the victim to a phishing site while still appearing to complete a real BharatHunt sign-in.
- Suggestion: Accept only relative in-app paths: must start with `/`, must not start with `//`, reject `\` and encoded variants. Apply the same allowlist when reading `sessionStorage` and before `router.replace` / `window.location.replace`. Prefer `router.replace` for validated relative paths only.
- Status: open

### Issue 2 -- Severity: bug
- File: `packages/backend/lib/services/authService.ts:271-288`
- File: `packages/backend/lib/routes/authRoutes.ts:126-133`
- File: `packages/frontend/lib/auth/AuthContext.tsx:58-81`
- Description: Refresh uses strict rotation (delete old token, then issue a new one). Concurrent `/api/auth/refresh` calls with the same cookie race: the loser fails, and the route’s `catch` always `clearRefreshCookie`. That clear can wipe the winner’s newly set cookie. Triggers include multiple tabs bootstrapping at once and React 18 Strict Mode double-mounting `useEffect` in development (Next defaults Strict Mode on). Result: intermittent forced logout right after a valid refresh.
- Suggestion: Serialize refresh per token (transaction / `UPDATE … WHERE token = ? RETURNING` / single-use claim), or tolerate reuse within a short grace window (token family). On the client, dedupe in-flight refresh (module-level promise) so Strict Mode and multi-mount only hit the network once. Only clear the cookie on definitive auth failures (unknown/expired token), not on every error path if that can collide with a concurrent success response.
- Status: open

### Issue 3 -- Severity: bug
- File: `packages/frontend/lib/api.ts:24-65`
- File: `packages/frontend/lib/auth/AuthContext.tsx:58-81`
- File: `packages/backend/lib/config/index.ts:20` (`JWT_EXPIRES_IN` default `15m`)
- Description: Access tokens live only in memory and are refreshed solely on AuthProvider mount via `/auth/refresh`. There is no 401 → refresh → retry path in `request()`. After ~15 minutes of SPA use without a full remount, `isAuthenticated` stays true (stale in-memory JWT + user) but authenticated API calls fail with 401 until the user reloads. That breaks the “session persists” acceptance bar for real product actions (vote, launch, etc.).
- Suggestion: On `401`/`UNAUTHORIZED` (excluding the refresh/login endpoints themselves), single-flight call `refreshSession()`, update the token store, and retry the original request once. Optionally proactively refresh slightly before JWT expiry using `exp` from the access token.
- Status: open

### Issue 4 -- Severity: bug
- File: `packages/backend/lib/services/authService.ts:147-155`
- File: `packages/backend/prisma/schema.prisma:95-104`
- Description: Refresh tokens are generated as high-entropy secrets but stored and looked up in plaintext (`token` column). Any DB leak (backup, SQL injection elsewhere, compromised Supabase credentials, overly broad service role) immediately yields reusable session secrets until expiry/rotation. Industry practice for long-lived bearer secrets is store only a hash (as with passwords).
- Suggestion: Store `sha256` (or HMAC with `REFRESH_TOKEN_SECRET`, which is already required in config but unused) of the token; look up by hash; keep only the raw value in the HttpOnly cookie. Migrate by issuing new tokens on next refresh.
- Status: open

### Issue 5 -- Severity: bug
- File: `packages/backend/lib/services/authService.ts:286-288`
- Description: Rotation deletes the existing refresh row *before* `issueSession` creates the replacement. If `create` fails (DB blip, constraint, process crash), the user has no valid refresh token left and must re-authenticate. Combined with Issue 2, error handling is unforgiving.
- Suggestion: Perform rotation in a single transaction: insert new token, delete old token (or atomic update of the token hash). On failure, roll back so the old token remains valid.
- Status: open

### Issue 6 -- Severity: suggestion
- File: `packages/backend/lib/routes/authRoutes.ts:16-24`
- Description: Cookies use `sameSite: 'strict'`. That is correct when frontend and API share a site (e.g. `app.example.com` + `api.example.com`). Decision log targets Vercel + Cloud Run; default hosts (`*.vercel.app` vs `*.run.app`) are cross-site. Browsers will not send Strict cookies on cross-site `fetch`, so PKCE verifier and refresh cookies will never round-trip in that setup — OAuth completion and session restore break in production until custom same-site domains are used.
- Suggestion: Document required same-site deployment topology. If cross-site hosting is required, use `sameSite: 'none'` + `secure: true` and ensure CSRF posture (Origin checks on cookie-authenticated POSTs). Prefer shared parent domain + `sameSite: 'lax'` or `'strict'`.
- Status: open

### Issue 7 -- Severity: suggestion
- File: `packages/backend/lib/routes/authRoutes.ts:67-70`
- File: `packages/backend/lib/middleware/rateLimiter.ts:19-25`
- Description: `authLimiter` wraps the entire `POST /login` surface, including OAuth start, magic-link send, *and* code / tokenHash / accessToken completion. A user who burns five attempts (typos, OTP retries, double-clicks) can then complete provider redirect and get `429` instead of a session — a poor failure mode after a successful Google/GitHub consent.
- Suggestion: Rate-limit start modes (`provider`, `email`) more aggressively; leave completion modes on a separate, higher limit (or keyed by code/token identity). Keep global abuse protection via `apiLimiter`.
- Status: open

### Issue 8 -- Severity: suggestion
- File: `packages/backend/lib/config/index.ts:21`
- Description: `REFRESH_TOKEN_SECRET` is validated as required (`min 32`) but never referenced. Operators must provision a secret that currently does nothing; it also suggests hashing/HMAC was planned (see Issue 4) and left incomplete.
- Suggestion: Use it to HMAC-hash refresh tokens before storage, or remove it from the env schema until needed so config matches behavior.
- Status: open

### Issue 9 -- Severity: suggestion
- File: `packages/backend/lib/middleware/auth.ts:47-69`
- File: `packages/backend/lib/routes/productRoutes.ts:13-22`
- Description: `optionalAuthMiddleware` correctly attaches `req.user` when a valid Bearer token is present and is unit-tested, but no route mounts it. Public product list/detail therefore cannot populate `hasVoted` (mapper already supports the field; list handlers ignore auth).
- Suggestion: When vote state is implemented on public reads, mount `optionalAuthMiddleware` on `GET /products` and `GET /products/:slug` and pass `req.user?.id` into the service. Not blocking for pure auth AC, but the middleware is otherwise dead code in the running app.
- Status: open

## What looks solid (no issue filed)

- OAuth PKCE: verifier generated server-side, stored HttpOnly, exchanged with code — good pattern for backend-mediated Supabase OAuth.
- Access JWT never in `localStorage` / `sessionStorage`; only memory via `tokenStore`.
- Refresh cookie flags: `httpOnly`, `secure` in production, path `/`.
- Logout deletes DB refresh row and clears cookie; client clears memory token in `finally`.
- Login body schema enforces exactly one mode (`provider` | `email` | `code` | `accessToken` | `tokenHash`).
- CORS credentials limited to configured frontend origin; Helmet enabled.
- `authMiddleware` / `optionalAuthMiddleware` unit coverage for missing, non-Bearer, expired, malformed, wrong-secret, and optional pass-through cases.
- `authService` unit coverage for OAuth URL, magic link, code exchange, access token, token hash, refresh rotation, logout.
- Nav logout and AuthProvider bootstrap paths are wired; callback handles `code`, `token_hash`, and hash-fragment `access_token`.
